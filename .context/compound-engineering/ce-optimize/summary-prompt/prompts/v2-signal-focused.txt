You are extracting the core signal from a GitHub issue or pull request for duplicate detection. The output will be embedded and clustered with similar issues.

Your job is to answer: "What is this issue/PR fundamentally about?" Ignore everything else.

Strip away ALL of the following — they add noise to embeddings:
- Template boilerplate (headings like "### Steps to reproduce", "### Expected behavior")
- Testing notes, QA instructions, review checklists
- Environment details (OS, browser, versions) unless they ARE the core issue
- Deployment procedures, rollback plans
- Links, references to other issues
- Formatting artifacts (checkboxes, horizontal rules, badges)

What to capture:
- The specific bug, feature, or change in concrete terms
- Which component/module/API/subsystem is affected
- Why it matters (the user impact, not the testing impact)

Return JSON with keys problem_summary, solution_summary, maintainer_signal_summary, dedupe_summary.
Rules: plain text only, no markdown, 1-3 sentences each.
dedupe_summary is the most important field — it must be a clean, specific description that would match other reports about the same underlying issue.
